# I18n Development Guide

## Quick Rules

- Add or change language metadata only in
  `src/shared/i18n/contract/locales.json`.
- Run `pnpm run i18n:generate` after editing the contract.
- Put cross-surface stable terms in
  `src/shared/i18n/resources/shared/<locale>/terms.json`.
- Keep surface copy in the surface resource owner.
- Store translation resources as readable UTF-8 text. Use escapes only for
  characters that must be escaped, such as control characters, quotes,
  backslashes, or explicit regex/codepoint ranges.
- Do not import `src/web-ui/src/locales` from `src/mobile-web`,
  `BitFun-Installer`, Rust crates, or server apps.
- Persist and send canonical app locale ids, not aliases.

## Commands

```bash
pnpm run i18n:generate
pnpm run i18n:contract:test
pnpm run i18n:audit
```

Run the nearest surface checks after changing resources:

```bash
pnpm run type-check:web
pnpm --dir src/mobile-web run type-check
pnpm --dir BitFun-Installer run type-check
cargo check --manifest-path BitFun-Installer/src-tauri/Cargo.toml
cargo test -p bitfun-core i18n -- --nocapture
```

## Where To Put Strings

Use shared terms only for labels that must stay semantically aligned everywhere,
such as product names, feature names, mode names, core agent names, common tool
names, and connection method labels.

Use surface resources for workflow text, button copy, validation messages,
empty states, toasts, page titles, and copy that is specific to a product
surface.

## Key Policy

Use stable, semantic keys. Do not encode current English copy in the key.

Good:

```text
sessions.startRemoteFlow
connectionMethods.bitfunServer
tools.explore
```

Avoid:

```text
clickHereToStartANewRemoteFlow
button1
newFeatureText
```

Shared term keys are reserved for concepts that are expected to mean the same
thing across surfaces:

- `product.*`
- `features.*`
- `modes.*`
- `agents.*`
- `tools.*`
- `connectionMethods.*`
- `statuses.*`

Do not put long sentences, layout-specific copy, or flow-specific copy in shared
terms. Those belong in the surface resource owner.

Do not rely on literal fallback strings in component code. Add the missing key
to the relevant locale file instead. Literal fallback strings make audits and
translation completeness harder to enforce.

## Loading Size

Unifying the language contract must not unify all translation bundles. Each
surface should ship only the resource set needed for that surface and the current
locale path.

For bundle-sensitive changes, compare generated asset sizes before and after the
change and include the result in the PR.

## Language Detection

Input boundaries may accept aliases from browsers, old configs, and links.
Convert them to canonical ids immediately:

- Frontend: generated `resolve*Language` or `resolveLocaleId` helpers.
- Backend: generated contract helpers or the i18n service registry.

Persisted values must use canonical ids.

## PR Expectations

Every i18n PR should state:

- Which surfaces changed.
- Whether the contract changed.
- Whether generated files were updated.
- Which i18n verification commands passed.
- Whether any surface intentionally does not support a new locale.
